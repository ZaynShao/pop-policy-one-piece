## 0. 元学习核心观察

AI 在最小 brief(仅产品一句话 + 5 角色名称 + 四级行政区划)下,**高概率往"常规政企 CRM"方向靠**:Opportunity(机会)+ Engagement(接触)+ Policy 建档 + SupportRequest(支援请求)+ Campaign(跨区域战役)+ 关系活跃度评分。

这个"靠"的方向不是无来由的 —— brief 中的"政府事务 + 属地拓展 + 工具集 + 热力图"确实很像"政企 CRM 热力图版"。但**用户版的实际方向大幅偏离这条默认路径**:

- **废弃政策建档** → 政策主题 + 外部工具返回覆盖清单
- **废弃中台队列** → 中台的"响应一线"在系统外完成(IM / 电话 / 邮件)
- **决策层非系统用户** → 由负责人线下汇报
- **工具体系作为核心差异化** → 成品文档 + 调用接口 + 级联匹配
- **地图元素作为一等交互**(图钉 / 蓝点 / 拜访点 + 涂层)—— 不是"地图是 CRM 的可视化外皮",而是"地图本身是工作台"

这五处偏离构成了用户版的"反 CRM 设计哲学"。AI 版的价值在于**证明了这条偏离是主动选择,不是无意识** —— 如果让普通 AI / 市面上大部分 PRD 模板,都会长成 AI 版的样子。

## 2. D2 · 角色解读差异(重要)

| 角色 | AI 版 | 用户版 | 分歧判定 |
|---|---|---|---|
| GA 负责人 | 看全国盘 + 定方向 + 加码决策 | 战略对齐 + 政府高层亲对接 + **对决策层线下汇报** | 基本一致,但用户版强调"对外" |
| PMO | **项目管理中枢**(任务池 + 战役 + 报表生成) + 独立战略功能 | **负责人的一双手**(操作执行,不做独立战略判断),与负责人权限等同 | ❌ **重大分歧** |
| 属地 GA | **数据域按省/市硬隔离**,跨域协作需临时授权 | **不做地域硬隔离**,主责由组织流程/OKR 决定,系统不限 | ❌ **重大分歧** |
| 中台 GA | 政策解读 + 支援请求响应 + 知识支持 | 政策主题维护 + **工具供应**(文档+接口) + 监管分发 | 中等分歧(核心产出物不同) |
| 决策层 | **系统用户**(只读 + 极简视图 + 高管视图) | **非系统用户**,由负责人线下汇报 / 导出截图服务 | ❌ **重大分歧** |

### 重大分歧详述

1. **PMO 定位**:AI 版把 PMO 做成"独立的项目管理+战略中枢";用户版明确 PMO 是"操作执行,常与负责人一人兼任"。这影响 PMO 的首屏、独立工作台、是否有独立战略决策权。
2. **属地 GA 硬隔离**:AI 版假设"驻点即权限边界";用户版 2026-04-23 拍板"系统不做硬隔离,人是主责实体"。AI 版的这个揣测来自传统 CRM 的"地盘"思维,但用户实际组织架构更灵活(业务划分交叉)。
3. **决策层角色**:AI 版默认"高管愿意亲自打开系统看";用户版拍板"决策层不上系统"。AI 版自己的 §附录判断点 #4 也承认这条可能是错判,值得尊敬。

## 4. D4 · 字段命名 / 颗粒度差异

**核心实体命名重合度 ≈ 0**。双方对"同一业务单位"给了完全不同的名字:

| 业务语义 | AI 版命名 | 用户版命名 |
|---|---|---|
| 一个重点项目/合作机会 | Opportunity | Pin(图钉) |
| 一次拜访 | Engagement(type=拜访) | Visit |
| 政府对接人 | GovContact(独立档案) | Visit.contact_person(字段级) |
| 政府机构 | GovOrg(独立档案) | Visit.department(字段级) |
| 政策本体 | Policy(一等实体) | PolicyTheme + CoverageItem(分两层) |
| 任务 | Task(一等实体) | PlanPoint(蓝点,粒度不同) |
| 留言 | Comment | Comment(一致,但 AI 版允许编辑,用户版不可改不可删) |
| 工具包 | 无(仅有 Policy 附件概念) | Tool + DocumentTool + InterfaceTool + ToolBinding + ToolConsumptionLog |

**颗粒度差异**:
- 政府端建档:AI 版深(机构/关键人/分层/偏好/活跃度);用户版浅(仅 Visit 字段)
- 政策建模:AI 版深(标题/文号/发文机关/解读状态机/传导链);用户版浅(主题+外部覆盖清单)
- 机会/项目:AI 版深(金额/把握度/战略相关度/阶段);用户版浅(Pin 标题+状态)
- 工具体系:用户版深(5 类相关实体);AI 版无

## 5. D5 · 状态机差异

| 实体 | AI 版状态数 | 用户版状态数 | 差异 |
|---|---|---|---|
| Policy / PolicyTheme | 5(新录入/解读中/已解读/已失效/已归档) | 3(draft/published/archived) | AI 版更细 |

| SupportRequest | 4 状态 + 驳回分支 | 无此实体 | |
| Tool | 无此实体 | 3(draft/published/archived) | |

AI 版状态机偏向"业务生命周期"(CRM 风),用户版偏向"地图元素可视化状态"(热力图风)。

## 6. D6 · 边界严格度

AI 版 0.3 明确列出 V1.0 不做:对外门户 / 自助 BI / 财务合同 / 政务系统对接 / 海外 GA / OA 通用办公

用户版 0.5 明确列出"不做":数据录入=系统的过度等号 / 政策一键落地幻想 / 数据沉淀=知识生产

**差异**:
- AI 版边界偏"功能模块层面"(哪些功能不做)
- 用户版边界偏"设计理念层面"(哪些思维误区不陷入)
- 两者不冲突,可以合并为"边界 + 边界的动机"

## 7. D7 · 非功能差异

| 项 | AI 版 | 用户版 |
|---|---|---|
| 热力首屏 | ≤ 2 秒 | ≤ 3 秒 |
| 可用性 | **99.9%**(9-19 时) | MVP 不定 SLA |
| 敏感字段 | **字段级权限**(GovContact.phone/wechat/preference_notes) | 无字段级;导出级水印+可选脱敏 |
| 弱网 / 离线 | 接触记录本地暂存(P1) | 未展开 |
| 批量导入 | 单次 500 条 ≤ 10 秒 | 未定 |
| 埋点 | 前端性能 + 业务事件 | 偷懒写"5xx 率 1% 告警" |

AI 版非功能更具体,用户版偷懒。AI 版的"99.9% SLA"对内部工具偏激进。

## 8. D8 · 盲点互补(值得用户关注)

### AI 版补出,用户版未考虑(候选采纳清单)

1. ⚠️ **Campaign 跨区域战役建模** —— 用户版 1.3 已拍"系统不做地域硬隔离,跨地域协作是常态",但没对"跨地域协作项目本身"建模。AI 版的"虚拟项目组"概念可能是盲点。**建议评估:要不要在第 8 章或 PRD-comparison.md 阶段拉进来。**
2. ⚠️ **GovOrg / GovContact 关键人档案** —— 用户版仅在 Visit 里记录对接人,没独立档案。如果业务真的看重"关系经营"(政府关键人离职/调岗跟踪),这是缺失。**但用户版场景 1/2/3/4/5 都没提"查对接人档案"的动作**,可能确实不需要。
3. ⚠️ **字段级敏感字段保护** —— 用户版只有导出级水印/脱敏,没有数据库字段级权限(如:phone / preference_notes 只让 owner 看)。如果真做关键人档案,字段级权限是必需的。
4. ⚠️ **异动告警** —— 热力某省大幅变化自动告警。用户版 F2 监管风险图层有类似思想但仅限监管事件。更广义的"异动自动提醒"值得考虑。
5. **月度/季度报告一键生成** —— 用户版 F7 是周报 AI 生成(远期)。AI 版把月报 P1,贴近 PMO 核心诉求。**用户版 1.2 PMO 职责有"协助导出材料给决策层",可能应该把月报提到 P1 而不是 F7**。
6. **时间轴回放** —— 过去 N 月态势演化动画。用户版无。UX 价值,但非 MVP。

### 用户版有,AI 版漏掉(用户版真正的差异化)

1. 🎯 **工具体系(成品文档 + 调用接口 + 级联匹配)** —— 这是用户版的核心创新。AI 版完全没想到"工具包"这种东西,默认是"政策解读 + 支援请求"的传统路径。
2. 🎯 **涂层渲染 + 1 主题=1 涂层** —— 地图可视化的 UX 创新。AI 版只有"热力维度切换",更像仪表盘套壳。
3. 🎯 **蓝点 → 红黄绿点的拜访转色流程** —— 将"计划"和"执行"绑到地图元素。AI 版用 Task 实体处理但没这个地图语义。
4. 🎯 **政策主题 + 外部工具返回覆盖清单** —— 废弃"录入政策"转向"订阅政策分析结果"。AI 版走经典录入路线。
5. 🎯 **决策层非系统用户 + 导出服务** —— 避开"让高管打开系统"的伪需求。AI 版自己的 §附录判断点 #4 也承认这条可能错判。
